# Edging Trainer V2 Roadmap

更新时间：2026-04-08

这份路线图不是简单地罗列待办，而是把接下来几轮工作的先后关系、依赖关系和验收边界写清楚。现在项目已经跨过“是否能跑起来”的阶段，真正重要的是怎样把视觉体验、奖励机制、长期成长判定、云端同步和公开发布逐步推进到一个彼此兼容、不会互相拖累的状态。

## Phase 1：工程清理与联调收口

这一阶段的目标，是先让代码树、构建链和运行边界彻底清楚。源码树里那批 `.ts/.tsx` 对应的 `.js` 镜像文件已经追查过并清理掉，根因也已经确认是构建脚本直接执行 `tsc` 导致输出写回 `src`；现在这部分已经改成只做类型检查，不会继续污染源码树。于是这一阶段剩下的重心，就更明确地落在 Supabase 联调、RLS 校验和真实同步流程上。

这一阶段同时要完成 Supabase 的第一次真实联调，包括 `.env.local` 配置、数据库 migration 执行、邮箱魔法链接登录、自有数据同步、公开排行榜视图读取，以及 RLS 是否真的把敏感字段挡在公开层之外。等这一部分跑通以后，项目才算真正从“本地功能版”进入“可发布候选版”。

交付标准：

1. `src` 源码树不再出现来源不明的 `.js` 镜像。
2. `npm run test -- --run` 和 `npm run build` 持续通过。
3. Supabase 登录、同步、公开榜单与 RLS 校验完成一轮手动验证。

## Phase 2：奖励系统深化

第二阶段围绕勋章系统继续往前推进，因为奖励感是这个产品最重要的叙事轴。现在勋章家族、材质等级和页面骨架已经存在，下一步要让它们从“分类齐全的奖励卡片”变成真正有收藏冲动和晋升感的目标系统。

这意味着三类工作要一起推进。首先是视觉层，需要把金属材质、边缘高光、压印中心纹章、等级差异、解锁动画和铭带文案做得更有层次；其次是规则层，要把每一枚勋章的解锁阈值写得更透明、更均衡，并让它们和长期成长算法形成稳定呼应；最后是叙事层，要让复盘页、勋章页和训练结束弹层都明确告诉用户“你为什么拿到这枚勋章”、“下一枚还差多少”、“这一枚代表的是节律、控稳还是耐力上的推进”。

交付标准：

1. 勋章家族与四级材质的视觉差异足够明显，不再只是换一层颜色。
2. 每枚勋章都能在 UI 中解释其解锁原因、当前进度与下一步目标。
3. 勋章解锁在训练结束、复盘页和个人公开资料之间保持同一套语言。

## Phase 3：长期成长算法与天梯校准

第三阶段的重点，是把“感觉上像一个成长系统”的东西，进一步推到“逻辑上可以自洽解释”的程度。现在长期成长已经采用 `14 天 / 56 天 / 120–180 天` 的窗口结构，天梯分数也已经按 `55% mastery + 30% growth + 15% consistency` 组织起来；接下来真正要做的，是对阈值、归一方式、置信度折扣和赛季段位边界做校准。

这里最值得认真投入的，不是去追求某种伪装成绝对科学的评分，而是保证算法有几条清楚的性质：它确实会奖励更短的休息、更长的持续刺激、更高的控制和更低的波动；它不会因为一次极端样本就把长期评价拉得过头；它对样本不足会坦率承认不确定性，而不是假装精确。等这些性质在测试和样本回放里都站得住，天梯才会真正有可信度。

交付标准：

1. 长期成长、历史锚点、置信度折扣与赛季分段都有测试覆盖。
2. 至少准备一批人工样本，验证“短休息、长持续、低波动”会被算法稳定奖励。
3. 天梯页能解释当前分数来自哪三部分，以及为什么最近发生了上升或回落。

## Phase 4：移动端与产品壳层打磨

第四阶段的工作，是把“已经可以在手机上打开”推进到“真的像一款手机产品”。PWA 基础现在已经存在，但这还只是壳层的起点。接下来要继续优化底部导航的手指触达范围、复盘页卡片在窄屏上的阅读节奏、日历和详细记录的折叠方式、弹层在安全区内的表现，以及动画在低端设备上的节制策略。

除此之外，训练页作为最高频场景，还值得进一步强化主操作区，把计时、阶段、节律提示和主按钮真正收束成一个视觉中心。这样做的好处很明显，因为训练页越像一个完整的“主舞台”，勋章与复盘页承接起来就越自然。

交付标准：

1. 常见手机视口下没有明显遮挡、压缩或跳动。
2. 训练、复盘、勋章、天梯四页在手机上都能单手完成主要动作。
3. Playwright 至少覆盖一条完整的手机流程。

## Phase 5：GitHub 发布与公开部署

最后一阶段，才是真正面向外部的发布整理。这里不仅是把仓库推到 GitHub，也包括把 README、环境变量说明、数据库迁移说明、测试方式、PWA 特性和隐私边界都写明白。因为这个项目里既有公开排行榜，也有私有原始训练数据，所以“如何公开”和“公开到什么程度”必须在仓库文档里讲清楚。

部署层面可以优先考虑 Vercel 或 Netlify；如果 Supabase 与环境变量都已经稳定，静态前端的部署其实并不复杂。更重要的是在首次公开之前，先确认测试、构建、RLS、匿名访问策略和公开资料页展示字段都已经达成一致。

交付标准：

1. GitHub 远端已绑定，首个版本完成提交与推送。
2. CI 在远端仓库中可正常跑通。
3. 有一份明确的公开部署记录，包含环境变量、数据库迁移和回归检查。

## 当前最值得优先处理的三件事

1. 先把 Supabase 联调真正走通，因为这一部分决定了天梯是否能从本地演示进入真实跨用户结构。
2. 继续深化勋章系统，因为它是这个产品最强的驱动机制，也是 UI 审美最能显著提升产品感的一层。
3. 尽快完成 GitHub 远端绑定与第一次正式推送，这样后面的 CI、部署和回溯记录才有稳定依托。
